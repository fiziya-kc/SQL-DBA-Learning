Stored Procedures

Precompiled SQL code stored in the database for reusable, secure, and efficient data operations.

What is a Stored Procedure?

A stored procedure is a precompiled collection of SQL statements stored in the database. It can accept input parameters, perform operations, and return results. Stored procedures are compiled once and stored in cache for repeated execution.

Benefits of Stored Procedures

  • Performance - Precompiled and cached by SQL Server
  • Security - Prevents SQL injection, controls data access
  • Reusability - Write once, use many times
  • Maintenance - Logic in one place, easy to update
  • Reduced Network Traffic - Execute complex logic server-side
  • Transaction Control - Can wrap multiple operations in a transaction

Basic Stored Procedures

-- Simple stored procedure with no parameters
CREATE PROCEDURE usp_GetAllEmployees
AS
BEGIN
    SELECT employee_id, first_name, last_name, salary
    FROM employees;
END

-- Execute the procedure
EXEC usp_GetAllEmployees;

-- Stored procedure with input parameters
CREATE PROCEDURE usp_GetEmployeesByDepartment
    @DepartmentID INT
AS
BEGIN
    SELECT first_name, last_name, salary
    FROM employees
    WHERE department_id = @DepartmentID;
END

-- Execute with parameter
EXEC usp_GetEmployeesByDepartment 10;
EXEC usp_GetEmployeesByDepartment @DepartmentID = 20;

OUTPUT Parameters

-- Stored procedure with OUTPUT parameter
CREATE PROCEDURE usp_GetEmployeeCount
    @DepartmentID INT,
    @EmployeeCount INT OUTPUT
AS
BEGIN
    SELECT @EmployeeCount = COUNT(*)
    FROM employees
    WHERE department_id = @DepartmentID;
END

-- Call with output parameter
DECLARE @Count INT;
EXEC usp_GetEmployeeCount 10, @Count OUTPUT;
SELECT @Count AS EmployeeCount;

-- Multiple OUTPUT parameters
CREATE PROCEDURE usp_GetDepartmentStats
    @DepartmentID INT,
    @EmployeeCount INT OUTPUT,
    @AvgSalary DECIMAL(10,2) OUTPUT
AS
BEGIN
    SELECT 
        @EmployeeCount = COUNT(*),
        @AvgSalary = AVG(salary)
    FROM employees
    WHERE department_id = @DepartmentID;
END

INSERT, UPDATE, DELETE

-- Insert with stored procedure
CREATE PROCEDURE usp_InsertEmployee
    @FirstName NVARCHAR(50),
    @LastName NVARCHAR(50),
    @DepartmentID INT,
    @NewID INT OUTPUT
AS
BEGIN
    INSERT INTO employees (first_name, last_name, department_id)
    VALUES (@FirstName, @LastName, @DepartmentID);
    
    SET @NewID = SCOPE_IDENTITY();
END

-- Update with stored procedure
CREATE PROCEDURE usp_UpdateSalary
    @EmployeeID INT,
    @NewSalary DECIMAL(10,2)
AS
BEGIN
    UPDATE employees
    SET salary = @NewSalary
    WHERE employee_id = @EmployeeID;
END

-- Delete with stored procedure
CREATE PROCEDURE usp_DeleteEmployee
    @EmployeeID INT
AS
BEGIN
    DELETE FROM employees
    WHERE employee_id = @EmployeeID;
END

Modifying & Dropping Procedures

-- Modify (recreate) stored procedure
ALTER PROCEDURE usp_GetAllEmployees
AS
BEGIN
    SELECT employee_id, first_name, last_name, salary, hire_date
    FROM employees
    ORDER BY hire_date DESC;
END

-- Drop stored procedure
DROP PROCEDURE usp_GetAllEmployees;

-- View procedure definition
EXEC sp_helptext 'dbo.usp_GetAllEmployees';

-- View procedure information
EXEC sp_help 'dbo.usp_GetAllEmployees';

-- List all procedures
SELECT name, create_date, modify_date
FROM sys.procedures
ORDER BY name;

Naming Convention: Use prefixes like 'usp_' for user stored procedures. This helps distinguish them from system procedures and makes them easier to find.
