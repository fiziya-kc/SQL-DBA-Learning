Database Constraints

Learn how to enforce data integrity rules at the database level using constraints.

What are Constraints?

Constraints are rules enforced on table data to maintain data integrity, accuracy, and reliability. They prevent invalid data from being entered into the database.

NOT NULL

Column cannot contain NULL values

UNIQUE

All values in column must be distinct

PRIMARY KEY

Unique identifier (NOT NULL + UNIQUE)

FOREIGN KEY

References another table

CHECK

Values must satisfy a condition

DEFAULT

Provides default value

Creating Tables with Constraints

-- Complete example with all constraints
CREATE TABLE products (
    product_id    INT IDENTITY(1,1) PRIMARY KEY,
    product_name  NVARCHAR(100) NOT NULL,
    sku           VARCHAR(50) UNIQUE,
    price         DECIMAL(10,2) NOT NULL CHECK (price > 0),
    stock_level   INT DEFAULT 0,
    category_id   INT REFERENCES categories(category_id),
    created_date  DATE DEFAULT CAST(GETDATE() AS DATE),
    CHECK (stock_level >= 0)
);

-- Named constraints for better error messages
CREATE TABLE employees (
    emp_id    INT IDENTITY(1,1),
    first_name NVARCHAR(50) NOT NULL,
    salary     DECIMAL(10,2),
    CONSTRAINT PK_emp_id PRIMARY KEY (emp_id),
    CONSTRAINT UQ_first_name UNIQUE (first_name),
    CONSTRAINT CK_salary_positive CHECK (salary > 0)
);

Adding Constraints to Existing Tables

-- Add NOT NULL to existing column
ALTER TABLE employees
ALTER COLUMN last_name NVARCHAR(50) NOT NULL;

-- Add UNIQUE constraint
ALTER TABLE employees
ADD CONSTRAINT UQ_email UNIQUE (email);

-- Add CHECK constraint
ALTER TABLE employees
ADD CONSTRAINT CK_hire_date CHECK (hire_date <= GETDATE());

-- Add DEFAULT constraint
ALTER TABLE employees
ADD CONSTRAINT DF_is_active DEFAULT 1 FOR is_active;

-- Add FOREIGN KEY constraint
ALTER TABLE orders
ADD CONSTRAINT FK_customer
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id);

Removing Constraints

-- Drop a constraint
ALTER TABLE employees
DROP CONSTRAINT CK_salary_positive;

-- Drop multiple constraints
ALTER TABLE employees
DROP CONSTRAINT PK_emp_id, UQ_email;

Best Practice: Always define constraints at table creation time. Adding constraints later can fail if existing data violates them. Always test with existing data before adding constraints.
