Why Data Types Matter

Choosing the correct data type is crucial for database performance, storage efficiency, and data integrity. The right data type ensures your database runs fast and your data remains accurate.

Critical: NEVER use FLOAT or REAL for storing monetary values. Use DECIMAL instead. Floating-point numbers can cause rounding errors that are unacceptable in financial calculations.

Numeric Data Types

INT / INTEGER

Whole numbers from -2,147,483,648 to 2,147,483,647. 4 bytes. Use for most IDs and counts.

BIGINT

Large whole numbers. From -9,223,372,036,854,775,808 to 9,223,372,036,854,775,807. 8 bytes.

SMALLINT

Small whole numbers from -32,768 to 32,767. 2 bytes. Use for limited range values.

TINYINT

Very small numbers from 0 to 255. 1 byte. Great for status codes and flags.

DECIMAL(p,s) / NUMERIC

Exact decimal numbers. p = precision (total digits), s = scale (decimal places). Use for money.

MONEY / SMALLMONEY

Currency values. MONEY: -922,337,203,685,477.5808 to 922,337,203,685,477.5807. 8 bytes.

FLOAT / REAL

Approximate numeric for scientific calculations. NOT for money! Can cause rounding errors.

BIT

Integer type that can take values 0, 1, or NULL. Great for boolean/toggle values.

-- Numeric examples
-- DECIMAL: DECIMAL(10,2) = 10 total digits, 2 after decimal
product_price   DECIMAL(10, 2)   -- 99999999.99 max
tax_rate        DECIMAL(5, 4)     -- 999.9999 max
-- INT with IDENTITY for auto-increment
customer_id     INT IDENTITY(1,1) PRIMARY KEY
-- BIT for boolean flags
is_active       BIT DEFAULT 1
is_deleted      BIT DEFAULT 0

String/Character Data Types

CHAR(n)

Fixed-length string. Always uses n bytes. Pad with spaces if shorter. Best when data is consistent length.

VARCHAR(n)

Variable-length up to n. Uses only what's needed. Standard choice for most text.

VARCHAR(MAX)

Variable-length up to 2GB. Use instead of deprecated TEXT. Stores large text documents.

NVARCHAR(n)

Unicode variable-length. Use for international/multilingual data. Each char = 2 bytes.

NVARCHAR(MAX)

Unicode large text. For storing large Unicode content like XML, JSON, or long text.

NCHAR(n)

Fixed-length Unicode. Always uses n*2 bytes. Useful for codes that are always same length.

-- String type examples
-- Fixed length for codes that are always same length
country_code    CHAR(2)          -- 'US', 'UK', 'DE' - always 2 chars
postal_code     CHAR(10)         -- Always padded to 10 chars

-- Variable length for names and descriptions
first_name      NVARCHAR(50)       -- Up to 50 Unicode chars
email           NVARCHAR(100)      -- Email addresses
comments        NVARCHAR(MAX)      -- Large text content

-- VARCHAR without MAX (specifies max length)
phone_number    VARCHAR(20)        -- Phone numbers
-- VARCHAR with MAX (large text storage)
article_content VARCHAR(MAX)       -- Blog posts, articles

Date & Time Data Types

DATE

Date only (no time). Format: 'YYYY-MM-DD'. Range: 0001-01-01 to 9999-12-31.

TIME

Time only (no date). Format: 'HH:MM:SS.nnnnnnn'. Range: 00:00:00.0000000 to 23:59:59.9999999.

DATETIME

Date and time. 1753-01-01 to 9999-12-31. Precision: 3.33ms. Legacy type - use DATETIME2 instead.

DATETIME2

Enhanced datetime. 0001-01-01 to 9999-12-31. Precision: up to 100 nanoseconds. Recommended over DATETIME.

DATETIMEOFFSET

DateTime with timezone offset. Preserves timezone info. Best for distributed systems.

SMALLDATETIME

Date and time. 1900-01-01 to 2079-06-06. Precision: 1 minute. Legacy type.

-- Date/Time examples
date_of_birth   DATE
start_time      TIME(0)              -- Time with 0 fractional seconds
created_at      DATETIME2(3)          -- With 3 digits of milliseconds
updated_at      DATETIME2 DEFAULT SYSUTCDATETIME()
-- Default values
created_date    DATE DEFAULT CAST(GETDATE() AS DATE)
modified_date   DATETIME2 DEFAULT GETDATE()

-- Date formatting and extraction
SELECT YEAR(order_date), MONTH(order_date), DAY(order_date)
SELECT DATEPART(year, order_date), DATEPART(quarter, order_date)
SELECT DATEADD(day, 30, order_date) -- Add 30 days
SELECT DATEDIFF(day, start_date, end_date) -- Days between

Binary & Other Data Types

BINARY(n)

Fixed-length binary data up to 8000 bytes. Good for hashes and fixed-length codes.

VARBINARY(n)

Variable-length binary up to n bytes. Use for images, files, encrypted data.

VARBINARY(MAX)

Large binary data up to 2GB. Use for storing documents, images, videos.

UNIQUEIDENTIFIER

GUID/UUID. 16-byte GUID. Use for distributed systems or replication keys.

-- Binary examples
icon_image      VARBINARY(MAX)           -- Store images
document_file   VARBINARY(MAX)           -- Store documents
password_hash   VARBINARY(64)            -- SHA-256 hash

-- GUID/UniqueIdentifier
row_guid        UNIQUEIDENTIFIER DEFAULT NEWID()

-- JSON data (SQL Server 2016+)
metadata        NVARCHAR(MAX)           -- Store JSON
-- Or native JSON type (Azure SQL)
config_data     JSON

-- XML data
xml_data        XML

Creating Tables with Data Types

CREATE TABLE customers (
    customer_id       INT IDENTITY(1,1) PRIMARY KEY,
    first_name       NVARCHAR(50)  NOT NULL,
    last_name        NVARCHAR(50)  NOT NULL,
    email            NVARCHAR(100) UNIQUE,
    phone            VARCHAR(20),
    date_of_birth    DATE,
    account_balance  DECIMAL(18,2) DEFAULT 0,
    is_active        BIT DEFAULT 1,
    created_at       DATETIME2 DEFAULT GETDATE(),
    modified_at      DATETIME2 DEFAULT GETDATE()
);

CREATE TABLE orders (
    order_id         INT IDENTITY(1,1) PRIMARY KEY,
    customer_id      INT NOT NULL REFERENCES customers(customer_id),
    order_number     VARCHAR(20) UNIQUE,
    total_amount     DECIMAL(18,2) NOT NULL,
    status           VARCHAR(20) DEFAULT 'Pending',
    order_date       DATETIME2(3) DEFAULT GETDATE()
);

Best Practices: Use NVARCHAR for user-facing text (handles international characters). Use appropriate lengths - too short limits data, too long wastes space. Always include DEFAULT values for audit columns.
