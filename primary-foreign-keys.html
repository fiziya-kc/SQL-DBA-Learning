Primary & Foreign Keys

Understanding the fundamental concepts for establishing relationships and maintaining data integrity.

Home > Learning Path > Database Design > Primary/Foreign Keys

Primary Key

A Primary Key is a column (or set of columns) that uniquely identifies each row in a table. Every table should have a primary key.

Primary Key Rules

  • UNIQUE - Must have different values for every row
  • NOT NULL - Cannot contain NULL values
  • Immutable - Should rarely or never change
  • Simple - Integer values are best (faster comparisons)
  • Single per table - Only one primary key per table
-- Primary Key with IDENTITY (auto-increment)
CREATE TABLE employees (
    employee_id INT IDENTITY(1,1) PRIMARY KEY,
    first_name  NVARCHAR(50) NOT NULL,
    last_name   NVARCHAR(50) NOT NULL
);

-- Composite Primary Key (multiple columns)
CREATE TABLE order_items (
    order_id   INT NOT NULL,
    product_id INT NOT NULL,
    quantity   INT NOT NULL,
    PRIMARY KEY (order_id, product_id)
);

-- Add Primary Key to existing table
ALTER TABLE products
ADD PRIMARY KEY (product_id);

-- Uniqueidentifier as Primary Key
CREATE TABLE sync_table (
    row_id UNIQUEIDENTIFIER PRIMARY KEY DEFAULT NEWID(),
    data   NVARCHAR(100)
);

Foreign Key

A Foreign Key is a column that references the Primary Key of another table, establishing a relationship between the two tables.

Foreign Key Rules

  • References the Primary Key of another table
  • Ensures referential integrity - no orphan records
  • Can be NULL (if relationship is optional)
  • Can have duplicate values (for many side of 1:N)
  • Values must match existing values in referenced table
-- Create tables with Foreign Key relationship
CREATE TABLE departments (
    dept_id   INT IDENTITY(1,1) PRIMARY KEY,
    dept_name NVARCHAR(50) NOT NULL
);

CREATE TABLE employees (
    emp_id    INT IDENTITY(1,1) PRIMARY KEY,
    emp_name  NVARCHAR(100) NOT NULL,
    dept_id   INT,  -- Foreign Key column
    FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
);

-- Foreign Key with ON DELETE/UPDATE actions
CREATE TABLE orders (
    order_id     INT IDENTITY(1,1) PRIMARY KEY,
    customer_id  INT NOT NULL,
    order_date   DATETIME2,
    FOREIGN KEY (customer_id) 
        REFERENCES customers(customer_id)
        ON DELETE NO ACTION       -- Prevent deletion
        ON UPDATE CASCADE          -- Update if PK changes
);

Referential Actions

When the referenced (parent) row is updated or deleted, you can specify what happens to dependent (child) rows:

Available Actions

  • NO ACTION - Raise error if child records exist (default)
  • CASCADE - Delete/update child records automatically
  • SET NULL - Set foreign key to NULL
  • SET DEFAULT - Set foreign key to its default value
  • RESTRICT - Prevent deletion/update of parent
-- CASCADE example (deleting employee deletes related orders)
CREATE TABLE orders (
    order_id    INT IDENTITY(1,1) PRIMARY KEY,
    customer_id INT NOT NULL,
    FOREIGN KEY (customer_id) 
        REFERENCES customers(customer_id)
        ON DELETE CASCADE   -- Delete orders when customer deleted
);

-- SET NULL example (when customer deleted, order has no customer)
CREATE TABLE orders (
    order_id    INT IDENTITY(1,1) PRIMARY KEY,
    customer_id INT NULL,
    FOREIGN KEY (customer_id) 
        REFERENCES customers(customer_id)
        ON DELETE SET NULL   -- Set to NULL instead of delete
);

Primary Key vs Unique Constraint

Comparison

  • Primary Key - One per table, cannot be NULL, creates clustered index by default
  • Unique Constraint - Multiple allowed, can be NULL, creates non-clustered index

Tip: Use IDENTITY columns (auto-increment integers) for primary keys in most cases. They're simple, fast, and guaranteed unique. Use composite keys only when a natural key makes sense.

Warning: Never use meaningful business data as a primary key (like SSN or email). These can change, creating update cascades. Use surrogate keys (ID) instead.
