Understanding JOINs

In relational databases, data is organized into multiple tables to reduce redundancy. JOINs allow you to combine rows from two or more tables based on a related column between them.

Types of JOINs

INNER JOIN

Only rows with matching values in BOTH tables

LEFT JOIN

ALL rows from left table + matching from right

RIGHT JOIN

ALL rows from right table + matching from left

FULL OUTER JOIN

ALL rows from BOTH tables (with NULLs for non-matches)

-- Sample tables for JOIN examples
-- employees: employee_id, first_name, last_name, department_id
-- departments: department_id, department_name, location

-- INNER JOIN: Only matching records
SELECT 
    e.first_name,
    e.last_name,
    d.department_name
FROM employees e
INNER JOIN departments d ON e.department_id = d.department_id;

-- LEFT JOIN: All employees, even without department
SELECT 
    e.first_name,
    e.last_name,
    d.department_name
FROM employees e
LEFT JOIN departments d ON e.department_id = d.department_id;

-- RIGHT JOIN: All departments, even without employees
SELECT 
    e.first_name,
    e.last_name,
    d.department_name
FROM employees e
RIGHT JOIN departments d ON e.department_id = d.department_id;

-- FULL OUTER JOIN: All from both tables
SELECT 
    e.first_name,
    e.last_name,
    d.department_name
FROM employees e
FULL OUTER JOIN departments d ON e.department_id = d.department_id;

-- CROSS JOIN: Cartesian product (all combinations)
SELECT 
    e.first_name,
    d.department_name
FROM employees e
CROSS JOIN departments d;

Visual JOIN Examples

JOIN with Multiple Tables

-- JOIN multiple tables in one query
SELECT 
    o.order_id,
    o.order_date,
    c.customer_name,
    e.first_name + ' ' + e.last_name AS sales_rep,
    o.total_amount
FROM orders o
INNER JOIN customers c ON o.customer_id = c.customer_id
INNER JOIN employees e ON o.sales_rep_id = e.employee_id
WHERE o.order_date > '2024-01-01'
ORDER BY o.order_date DESC;

-- JOIN with aggregation
SELECT 
    d.department_name,
    COUNT(e.employee_id) AS employee_count,
    AVG(e.salary) AS avg_salary
FROM employees e
INNER JOIN departments d ON e.department_id = d.department_id
GROUP BY d.department_name
HAVING COUNT(e.employee_id) > 5
ORDER BY avg_salary DESC;

Subqueries

A subquery is a query nested inside another query. They're powerful for filtering based on calculated values or performing complex data retrieval.

-- Subquery in WHERE clause (scalar result)
-- Find employees earning above average
SELECT first_name, last_name, salary
FROM employees
WHERE salary > (
    SELECT AVG(salary) 
    FROM employees
);

-- Subquery with IN (list of values)
-- Find employees in top 3 departments by headcount
SELECT first_name, last_name, department_id
FROM employees
WHERE department_id IN (
    SELECT TOP 3 department_id
    FROM employees
    GROUP BY department_id
    ORDER BY COUNT(*) DESC
);

-- Subquery in FROM clause (derived table)
-- Calculate department averages then find highest
SELECT dept_summary.department_name,
       dept_summary.avg_salary
FROM (
    SELECT 
        d.department_name,
        AVG(e.salary) AS avg_salary
    FROM employees e
    INNER JOIN departments d ON e.department_id = d.department_id
    GROUP BY d.department_name
) AS dept_summary
ORDER BY dept_summary.avg_salary DESC;

-- Correlated subquery (references outer query)
-- Find employees earning more than their department average
SELECT first_name, last_name, salary, department_id
FROM employees e
WHERE salary > (
    SELECT AVG(salary)
    FROM employees
    WHERE department_id = e.department_id
);

-- EXISTS / NOT EXISTS (efficient for existence checks)
SELECT first_name, last_name
FROM employees e
WHERE EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = e.employee_id
    AND o.total_amount > 10000
);

JOIN vs Subquery: When to Use Each

  • Use JOINs when you need columns from both tables in your result
  • Use Subqueries when you need to filter based on aggregate calculations
  • Use EXISTS for existence checks - it's faster than IN for large datasets
  • Use NOT EXISTS for "find rows with no match" queries
  • Use CTEs (Common Table Expressions) for complex multi-step queries

Performance Warning: Subqueries can be slower than JOINs, especially correlated subqueries that run for each row. Always test both approaches with large datasets. Use EXPLAIN or execution plans to compare performance.

Common Table Expressions (CTEs)

CTEs are temporary named result sets that exist within the scope of a single query. They make complex queries more readable.

-- Simple CTE example
WITH DepartmentStats AS (
    SELECT 
        department_id,
        COUNT(*) AS emp_count,
        AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
)
SELECT 
    d.department_name,
    ds.emp_count,
    ds.avg_salary
FROM DepartmentStats ds
INNER JOIN departments d ON ds.department_id = d.department_id
WHERE ds.avg_salary > 70000;

-- Multiple CTEs
WITH
HighEarners AS (
    SELECT *
    FROM employees
    WHERE salary > 80000
),
SalesHighEarners AS (
    SELECT *
    FROM HighEarners
    WHERE department = 'Sales'
)
SELECT first_name, last_name, salary
FROM SalesHighEarners
ORDER BY salary DESC;

-- Recursive CTE (hierarchical data)
WITH OrgHierarchy AS (
    SELECT employee_id, first_name, manager_id, 1 AS level
    FROM employees
    WHERE manager_id IS NULL
    
    UNION ALL
    
    SELECT e.employee_id, e.first_name, e.manager_id, oh.level + 1
    FROM employees e
    INNER JOIN OrgHierarchy oh ON e.manager_id = oh.employee_id
)
SELECT REPLICATE('    ', level - 1) + first_name AS name, level
FROM OrgHierarchy;

Pro Tip: Always use table aliases (like 'e', 'd') to make your queries more readable and avoid column name ambiguity. Use meaningful aliases: customers AS c, orders AS o, etc.
